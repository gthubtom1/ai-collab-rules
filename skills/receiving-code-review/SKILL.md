# 接收代码审查

## 概述

代码审查需要的是**技术评估**，不是情绪表演。

**核心原则：实施前先核实；假设前先发问；技术正确 > 社交舒适。**

## 响应流程

```
收到审查反馈时：
1. 读：完整读完，先不反应
2. 懂：用自己的话复述这条要求（或发问）
3. 核：对照代码库的真实情况核实
4. 评：对「这个」代码库技术上成立吗？
5. 答：技术性确认 或 带理由的反驳
6. 做：一次一项，每项都验证
```

## 禁止的回应

**绝不：**
- 「你说得太对了！」（表演式附和，违反铁律）
- 「好棒的建议！」「非常感谢反馈！」（空洞）
- 「我这就去实现」（核实之前就答应）

**应该：**
- 复述这条技术要求
- 提澄清问题
- 错的就带技术理由反驳
- 或者直接动手（行动 > 表态）

## 反馈不清楚时

```
若任一条不清楚：
  停 —— 先别实施任何东西
  对不清楚的条目要求澄清
原因：条目之间可能相关，半懂 = 做错。
```

例：让你「修 1-6」，你懂 1/2/3/6、不懂 4/5 → **错**：先做懂的、回头再问；**对**：「我懂 1/2/3/6，4/5 需要先澄清再动手」。

## 按来源区别对待

- **来自你的人类伙伴**：可信，理解后实施；范围不清仍要问；不表演附和。
- **来自外部审查者**：实施前先核——对「这个」代码库对吗？会不会破坏现有功能？当前实现是不是有原因？跨平台/版本都成立吗？审查者是否了解全貌？**存疑就带技术理由反驳**；无法核实就明说「没有 X 我没法核实，要我去查/问/继续？」

## 本项目集成说明

本 skill 对应规则 A5「独立验证」：收到任何审查/提醒后按此处理，杜绝「你说得对」式盲从，保证技术严谨。
