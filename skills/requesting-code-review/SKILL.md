# 请求代码审查

## 概述

派一个**代码审查子代理**，在问题级联放大之前抓住它。审查者只拿到「精心准备的产物 + 需求」，**绝不给它你这次会话的思路历史**——这样它聚焦于成果本身、不被你的思维带偏（防自我认同偏差），也省下你自己的上下文继续干活。

**核心原则：早审、勤审。**

## 何时请求审查

**必须：**
- 完成每个有意义的任务后
- 完成重要功能后
- 合并到 main 之前

**有价值（可选）：**
- 卡住时（换个新视角）
- 重构前（基线检查）
- 修完复杂 bug 后

## 怎么请求

1. **取 git 范围**：`BASE_SHA=$(git rev-parse HEAD~1)`（或 origin/main）、`HEAD_SHA=$(git rev-parse HEAD)`
2. **派 code-reviewer 子代理**，只给它：
   - 你做了什么（简述）
   - 它应该满足什么（计划/需求）
   - BASE_SHA → HEAD_SHA（要审的 diff 范围）
   - **不给**你的对话历史/思路
3. **按反馈行动**：
   - Critical（严重）→ 立刻修
   - Important（重要）→ 继续前修
   - Minor（次要）→ 记下来稍后
   - 审查者错了 → 带技术理由反驳（别盲从）

## 为什么独立

研究表明：当验证者被自己/生成者的思路「带偏」时，漏检率高、还会趋同。所以审查者**必须**拿到干净、独立的上下文，把产物当「陌生人写的不可信代码」来审。

## 本项目集成说明

本 skill 对应规则 A5「独立验证」与 C1 自检：在「声称完成 / 合并 / 上线」前调用，配合 verification-before-completion 双保险。
